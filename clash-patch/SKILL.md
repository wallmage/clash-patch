# Clash 补丁与诊断

## 必须遵守

1. **绝对不要退出、停止或重启 Clash 客户端。** 不得执行、建议或要求用户这样做。
2. 先只读检查，再修改。只按已保存用途档位切换 TUN 或 Clash 自己的系统代理；不得覆盖第三方代理，也不得切换订阅、代理组或节点。
3. 只处理 Clash 当前存储位置中的订阅。无法确认本地/iCloud 状态时停止，不猜。
4. 要求 Mihomo 1.19.27 或更高版本。找不到内核、版本过旧或 30 秒内没有响应时不修改。
5. 候选必须通过 YAML 重读、二次转换一致性检查和 Mihomo 校验；失败时保留原文件。
6. 所有用户消息使用简体中文，不显示订阅地址、密码、UUID、私钥、控制器密钥或完整节点地址。

开始前完整阅读 [references/patch-policy.md](references/patch-policy.md)。详细产品规则和全部状态以该文件为准。

## 模块选择

- **Patch 模块**：用户首次安装、改变用途档位或明确要求网络配置时使用；只应用满足该档位的最少改动。
- **Diagnostics 模块**：用户说网络不对劲、变慢、打不开、不稳定、分流异常或出现其他网络现象时使用。不能因为用户提到 Clash 就先运行补丁。

用途档位同时约束 Patch 和 Diagnostics。Diagnostics 每次开始都先读取本机保存的档位；没有时先完成用途选择。档位决定诊断目标、允许检查和修改的范围，以及修复后的回归验收。故障本身不能自动升档；只有用户明确增加用途时才改档并重新确定范围。

## 用途档位

先读取本机保存的档位：macOS 运行 `bash scripts/install_macos.sh --show-profile`；Windows 运行 `.\scripts\install_windows.cmd -ShowUsageProfile`。没有已保存档位时，修改前必须问：**“你使用网络代理主要用于哪些用途？”**

1. **档位 1｜普通浏览**：Twitter、Facebook、YouTube 等。给当前存储位置中的全部订阅安装共同国内域名直连基线，并确认 Clash 客户端的“设置为系统代理”已开启；不修改 TUN、IPv6、WebRTC、AI 分组、节点或订阅自动更新。用 Computer Use 测试国内站、Google、Twitter 和一个用户常用站点。
2. **档位 2｜海外 AI**：继承档位 1 的共同国内域名直连基线；ChatGPT、Codex、Gemini、Perplexity 等，但不用 Claude。开启 TUN，并关闭 Clash 客户端自己的系统代理开关，避免同一流量被 Clash 重复接管；这不是为了隐藏代理。除下述 AdGuard for Mac 已知兼容路径外，不得关闭或覆盖第三方代理或 PAC；不增加 WebRTC 或 AI 分组补丁。测试国内站、Google、Twitter、ChatGPT 和 Gemini。
3. **档位 3｜Claude/Claude Code**：先完成档位 2，再关闭自动更新并应用完整增强：①配置 TUN、DNS 劫持与严格路由；②关闭 IPv6；③国内 DNS 直连大陆加密解析器；④普通国外与 AI DNS 按出站分开；⑤补全或创建 AI 分组；⑥补全 AI 规则；⑦所有 UDP 经 AI 分组并以拒绝兜底，防止 WebRTC 直连；⑧完成分流、DNS 和两项 WebRTC 验证。说明全局 UDP 会影响 QUIC、游戏、语音和视频；建议 AI 分组选择台湾家宽，其次日本家宽，但不得自动切换节点。

向小白用户展示以上三项及改动差异。用户可以随时改档；升档时只补新增能力。从档位 3 降到档位 1 或 2 时，先运行安全卸载：macOS 用 `bash scripts/uninstall_macos.sh`，Windows 用 `.\scripts\uninstall_windows.cmd`；再保存并执行新选择。卸载只能撤销能确认属于本工具且未被继续修改的设置，不能覆盖后来产生的用户改动；无法安全恢复的旧增强继续保留并说明。Windows 卸载返回 `partial` 时，本次没有修改整批保护文件，必须保留旧档位且不得继续降档，也不得要求用户退出、停止或重启客户端。保存并执行选择：macOS 用 `bash scripts/install_macos.sh --profile N`，Windows 用 `.\scripts\install_windows.cmd -UsageProfile N`。用户明确表达 `Claude` 或 `Claude Code` 的网络配置需求时，直接选择或升级为档位 3；明确表达其他海外 AI 时选择或升级为档位 2；仅提到某网站故障仍进入 Diagnostics，不借机改档。

Diagnostics 按保存档位工作：三个档位都维护共同国内域名直连基线。档位 1 只保障国内直连与普通境外网站、系统代理，不检查或修改 TUN，不做 DNS 泄漏、WebRTC、AI 分组或 Claude 检查。档位 2 增加 TUN、普通海外 AI 与 Agent 联网，但仍不做泄漏、WebRTC 或 AI 分组检查。档位 3 才维护其余完整增强；任何可能影响共同网络路径的修复都要复测相关第三档保护，不能为解决一个网站而破坏 DNS、分流或 WebRTC 防护。

**解决原始问题是 Diagnostics 的结束目标。** Diagnostics 默认只读，只表示第一阶段先取证；只读检查是第一阶段，不是交付结果。诊断结论不是完成，解释、建议或状态说明不能代替行动。只要仍有本机可安全执行、属于用户要求和当前档位的下一步，就继续修复、复测、继续迭代。不得以“已经找到原因”“给出方案”“需要观察”代替本机能够完成的修复。

代理不请求用户确认诊断方案，也不把计划、可能性列表或“请自行操作”当成交付。只有工具明确要求的操作时确认、缺少密码/MFA/实体操作、修改会超出当前档位或授权、存在不可恢复的数据风险，或外部条件确实阻止继续时，才提出一个精确的阻塞问题；先完成全部不需要确认的准备，说明将要执行的单一动作与恢复方法，用户授权后直接继续。不得把“请回复继续”当成常规收尾。

有 Computer Use 且症状能在界面看到时，必须在修改前复现原始症状；命令行结果不能代替用户实际使用的应用。再判断影响范围，按现场选择浏览器、应用、系统、DNS、连接、分流、传输质量和日志证据；不是每次全部检查。预计超过 10 分钟且存在彼此独立的只读证据来源时，可以使用多个 Sub Agent 并行取证；主代理仍串行完成所有写入和判断。配置缺陷不等于故障原因，只有同一原始症状的单变量对照和独立证据同时成立，才能修改并下结论。

浏览器白屏、一直转圈或过很久突然显示时，必须把 DNS、主文档与关键资源的网络耗时、首次显示时间和加载结束时间分开记录。怀疑扩展、过滤器或共同网络组件时，除单站目标的开关对照外，还要测试至少两个健康对照：只有目标异常就修目标交互；健康对照也异常就修共同组件本身，不能继续逐站加例外。多个无关目标在内容开始传输前出现相同等待时，只盘点当前档位允许的接管层；档位 1 不检查 VPN/TUN。用时间线、日志与单变量切换证明重叠接管后再做职责分层，每层只保留一个透明接管者。无改善立即恢复原设置。不能因为单个目标的对照结果就停用或删除整个组件。

macOS 的档位 2、3 检测到 AdGuard for Mac 时直接检查其过滤模式；若它以 `Network Extension` 与 Clash TUN 重叠接管，就通过 Computer Use 在 AdGuard 自己的界面切到“自动代理”，保留 Clash TUN，并按详细策略复测。这是已知兼容路径，不是升档；Clash TUN 存在时不得把 AdGuard 改为 `Network Extension`。不得用 `networksetup` 或直接编辑设置，不得逐站添加例外，也不得停用、退出或卸载 AdGuard。禁止按应用调整 AdGuard 过滤范围。档位 1 和 Windows 不照搬该组合。仅检测到 AdGuard 不能证明当前故障由它造成；Diagnostics 仍用修改前后的原场景对照确认，无改善立即恢复。

### 已知故障快查

- **第一档已知故障：国内请求误走海外。** 国内网站或应用时好时坏，且目标属于国内数据集、实时连接却命中 `MATCH`/海外代理，而大陆 DoH 与直连稳定时，档位 1、2、3 都处理。把同一份受管 `cn.mrs` 写入全部订阅，让 DNS 与 `RULE-SET,...,DIRECT` 共用数据源；每天更新，安全更新后重应用，名称或缓存路径冲突时编号，不覆盖用户配置。用户显式域名规则和档位 3 AI 规则优先，受管规则位于宽泛规则前；不得给 Kimi、欧陆词典等单站加例外。诊断核对每份订阅的提供器、DNS/路由引用和实时 `DIRECT`；缺项就用已保存档位的 JSON 安装入口一次修复全部订阅，齐全但走错再查提供器、缓存和活动配置。当前无法复现时写“暂未复现”。
- **AdGuard 自动代理、证书故障与 Fake-IP 重用。** 日志出现 CFNetwork `uses proxy`、本机 AdGuard 端口、`-9802`/`CERTIFICATE_VERIFY_FAILED` 或浏览器关键资源证书错误时，不归入国内误分流。先检查系统代理所有权、PAC 查询中断和 Clash `ProxyConfigHelper` 活动；档位 2、3 只允许“Clash TUN 开、Clash 系统代理关、AdGuard 自动代理开”。Mihomo 使用 Fake-IP 时，把 AdGuard 故障日志中的目标域名和假地址，与 Mihomo 持久 Fake-IP 数据库只读快照的当前反向映射交叉核对；同一个地址现在映射到其他域名，才确认 **Fake-IP 被重新分配**。再三重确认当前 **Mihomo HTTP 代理端口**确由 Mihomo 监听，只通过 AdGuard 界面把 **AdGuard 出站代理**设为 HTTP、`127.0.0.1` 和该端口，让请求按域名交给 Mihomo；不得固定端口，不覆盖已有的非 Clash 出站代理。修改前记录全部原字段，原目标三次、至少三个无关目标、广告过滤、证书处理或当前档位能力任一失败，就恢复 AdGuard 原状态。不得点击“继续”，不添加 Apple/iCloud 或单站规则，不归咎于订阅节点，不全局关闭 HTTPS 过滤，也不改 Plist、PAC 或系统代理。禁止按应用调整 AdGuard 过滤范围；Clash TUN 存在时不得把 AdGuard 改为 `Network Extension`。

代理必须主动完成本机可完成且当前档位允许的检查。历史证据不足时建立有时间戳的观察窗口，不让小白用户决定该查哪个日志。一次失败不是停止条件：修复失败后恢复原状态、继续取证，再按新证据选择下一项单变量修复。两次假设或修改未改善原始症状后，必须做诊断重置：恢复全部未生效的试验，重列已确认、已排除和未知项；诊断重置后继续读取新证据，没有新的证据不得进行第三次修改。确认问题后执行最小、可恢复的修复，并用同一应用、目标和动作连续复测；涉及网络接管模式时还要验证当前档位要求的过滤、分流或安全能力。本机不能解决时，再查当前官方方案。只有原始症状和相关回归项通过，或出现上述真实阻塞，才向用户收尾。最终用简体中文先报告结果，再简要说明证据、已做修复、复测结果和仍未确认之处。单项 Clash 配置修复仍留在 Diagnostics，只复用 Patch 的备份、候选校验、刷新和失败恢复机制；只有档位 3 执行其余完整策略与 Patch 专用验收，普通浏览器或应用问题不得顺手改 Clash。

## 事故防线

- 保留 `default-nameserver` 和 `proxy-server-nameserver` 的安全用户值；字段缺失或属于旧版危险固定值时按策略迁移。不得把节点启动解析改成 `1.1.1.1` 或 `8.8.8.8`。把 `direct-nameserver` 统一设为策略中的大陆 IP DoH，并关闭 `direct-nameserver-follow-policy`，让已判定为 `DIRECT` 的国内域名获得大陆 CDN。不得使用明文 DNS、`system` 或 ECS 作为直连域名解析器。
- 不得安装永久监听、LaunchAgent、`WatchPaths`、计划任务或目录监听。安装与卸载时只清理能确认属于旧版 Clash Patch 的遗留监听。
- 每次写入配置前创建包含日期时间的版本化备份，保留修改前的版本；第一次运行另存初始快照。故障排查先做配置差异比较，需要回滚时先备份当前版本并校验预期 SHA-256。
- Windows 同一应用目录的公开操作由目录内独占锁文件串行执行，并在修改期间持有配置目录和目标父目录句柄；路径别名、目录改名或联接点替换不得产生第二个并发操作。批量文件事务开始前写入持久恢复记录；进程被强制结束后，下次操作必须先恢复原状态。备份目标身份使用相对应用目录的规范路径，不能因目录改名、大小写或扩展路径别名失效。
- Windows 通过已锁定原文件句柄写入替换内容或失败恢复内容时，必须先截为零长度再写入并同步；强制中断只能留下空文件或目标内容前缀，不能留下新内容与旧尾部拼接的不可恢复状态。
- Windows 新建目标必须在最终路径出现前发布私有准备记录；主事务记录发布前中断时，下次公开操作只删除记录中仍为空且核对期间未变化的新目标，遇到非空内容、链接或身份变化时拒绝继续。主事务发布并删除准备记录后才能写入新目标。安全更新验收失败时，整批订阅恢复与删除本次验收清单必须放进同一个可恢复事务；进程中断后不能把已经恢复的旧订阅误报为本次更新成功。
- macOS 必须把已有且有效的 REALITY `short-id` 保持为文本；不补齐、不截断、不猜测。Windows 脚本不改该字段。
- 任何候选都必须验证并再次转换；普通安装与安全更新必须先校验整批候选再写，任一份失败时不得留下部分改动。macOS 通过基础环境检查并建立初始快照后先保存用户明确选择的新档位，再修改自动更新和订阅；普通失败恢复旧档位，进程被强制结束则保留新档位作为恢复意图，下一次 Patch 或安全更新必须按它继续，不能读取旧档位降级处理。macOS 批量写入前保存持久事务记录，进程被强制结束时，下次 Patch 或安全更新先恢复未完成事务。Windows 自动查找 AppHome 时如果 Roaming 与 Local 候选同时存在，必须拒绝并要求明确指定当前目录，不能猜测选择。当前订阅只允许通过本地控制器自动刷新；刷新或运行检查失败时立即恢复原文件和原运行配置。档位 3 只在确实由本工具关闭自动更新时保存所有权状态；安全卸载把安装文件删除、自动更新恢复和所有权清理作为同一个可恢复事务，失败或中断时必须同时保留用途档位、关闭状态和所有权并报告。绝不退出 Clash。
- macOS 安全卸载在完整暂存并写出 READY 前不得删除目标；此前中断时丢弃不完整暂存。删除后的文件恢复必须以原子、不可覆盖的方式发布，恢复过程再次被强制结束时允许下次继续，不能留下会被误判为外部修改的半文件。
- macOS 普通安装、安全更新和安全卸载的完整公开流程必须共用同一个跨进程操作锁，从读取用途档位持续到最终提交。并发写操作等待最多 5 秒后返回 `operation_in_progress`，不能让卸载在最终核对后按路径删除另一个安装刚发布的新档位或所有权状态。卸载在删除后中断时，下一次安装或安全更新先在锁内完成未完成的安全卸载；恢复收据或状态无法确认时停止，不得绕过该事务写入新档位。
- macOS 当前订阅刷新成功后，必须通过本地控制器依次清除 Fake-IP 和 DNS 缓存，再做 DNS 与连接验证；缓存清理失败也要恢复原配置。
- macOS 普通 Patch 的未完成事务恢复不仅要还原文件；当前订阅可能已经加载候选时，必须先重新加载恢复后的原订阅并完成运行检查，成功后才能删除事务记录和处理新候选。强杀恢复或本次自动刷新回退中的运行恢复失败时，都要保留事务并停止；本次使用 `--no-reload` 时也必须保留事务并停止处理新候选。
- macOS 安全更新恢复事务时，必须从当前存储位置的全部配置识别活动项，不能只在远程订阅目标中查找。旧事务恢复和本次安全更新回滚都要等原配置重新加载并通过运行检查后才能删除事务；运行恢复失败时保留记录并停止下载或下一轮写入。
- `config.yaml` 是 ClashX Meta 的默认基础配置，不得删除；当前选择其他订阅时安静跳过它，不把它当成无效订阅报错。

## 配置历史、比较与恢复

第一次运行保存初始快照；以后每次实际写入前保存包含日期时间的修改前版本，不覆盖或自动删除旧备份。Windows 备份必须先在同目录临时文件中完整写入、同步并设置私有权限，再原子发布正式编号；中断的临时文件不能被列出、冒充初始快照或用于恢复。用户说某次改动后异常时，先列出备份，选择**症状出现前**最近的一份，再**先比较**、后判断；只报告变化字段、文件名和 SHA-256，不输出配置值。macOS 依次用 `--list-backups`、`--compare-backup ID`、`--restore-backup ID --expected-current-sha256 SHA256`；Windows 依次用 `-ListBackups`、`-CompareBackup ID`、`-RestoreBackup ID -ExpectedCurrentSha256 SHA256`。
只有配置差异与原始症状的证据相符、备份通过校验、目标仍是比较时的 SHA-256，才允许恢复。恢复前先备份当前版本，恢复后用原应用、原目标、原动作复测；无改善或验收失败时恢复回滚前版本。macOS 恢复备份前必须持有与 Patch、安全更新相同的操作锁并先处理未完成的配置事务；若事务恢复改变了当前字节，本次备份恢复按哈希冲突停止，重新比较后再试。macOS 恢复当前订阅后，必须通过本地控制器重新加载，保持恢复前的 TUN 开关和代理组选择，并完成缓存、DNS 与连接检查；任何一步失败都恢复回滚前版本。文件恢复但运行内核未恢复时必须单独报告，不能继续把原始症状复测写成已完成。恢复非当前订阅时只更新文件，不切换当前订阅。Windows 客户端正在运行时只做列出和比较，不停止客户端来强行恢复。

macOS 备份恢复遇到旧 Patch 事务时，必须在哈希判断前同时恢复文件和运行配置；运行恢复失败时保留事务并停止。备份文件替换前建立新的持久事务；文件已经与备份相同时，当前订阅的运行检查也必须有事务保护。运行加载与检查在同一操作锁和事务内完成；只有运行检查成功，或失败后文件与运行配置都已恢复，才能删除事务记录。

## 执行

| 环境 | 命令 |
| --- | --- |
| macOS + ClashX Meta | `bash scripts/install_macos.sh --profile N` |
| Windows + Clash Verge Rev | `.\scripts\install_windows.cmd -UsageProfile N` |
| 其他客户端 | 不修改；建议安装受支持的最新版客户端 |

三个档位的安装程序都检查 Mihomo，并把共同国内域名直连基线应用到当前存储位置中的全部订阅；档位 1、2 不增加 TUN、IPv6、WebRTC、AI 分组设置，也不改订阅自动更新。客户端开关由 Computer Use 操作并复测。Windows Computer Use 只能操作当前会话的前台桌面，实际工具不可用时不得假装已操作。档位 3 的安装程序额外关闭订阅自动更新且不依赖 Computer Use；macOS 当前订阅刷新后检查 TUN、DNS、外网连通性和原有代理组选择，失败时自动恢复。Windows 三档都使用全局扩展脚本，档位 1、2 只包含共同基线；已有脚本只有唯一同步 `main` 且没有其他 `main` 绑定时才能组合，否则整批不修改。两个平台都保持 Clash 运行。Skill 调用脚本时优先使用 JSON v1（macOS `--json`、Windows `-Json`），按稳定 `code` 与 `operation` 判断，不解析中文；输出不得含订阅 URL、密钥、完整路径或节点名称。

## 安全更新全部订阅

用户说“更新节点”“更新订阅”或同义表达时，进入安全更新，不把它当成只更新当前订阅：

1. 读取已保存档位和当前存储位置，列出全部远程订阅；不显示订阅地址。
2. 为每份当前配置创建带日期时间的修改前备份。macOS 运行 `bash scripts/install_macos.sh --safe-update`，一次下载全部远程订阅；全部下载、YAML 重读、二次转换和 Mihomo 校验成功后才统一替换。任何一份失败时尝试恢复全部文件；恢复本身失败必须逐份报告，不能误报全部保持原样。
3. 三个档位都对每份下载结果重新应用共同国内域名直连基线；档位 3 再应用完整补丁，档位 1、2 不增加第三档设置。更新后刷新当前订阅并复核运行状态，失败时整批回滚并重新加载原配置；订阅文件已恢复但运行内核未恢复时必须单独报告，不能写成全部保持原样。若上次安全更新在运行内核加载候选后被强制结束，下一次安全更新必须先恢复文件并重新加载恢复后的当前订阅；运行内核无法恢复时停止下载并报告待恢复。
4. Windows 先运行 `.\scripts\install_windows.cmd -SnapshotProfiles` 核对并备份全部远程订阅，再用 Computer Use 在 Clash Verge Rev 中执行“全部更新”，随后运行 `.\scripts\install_windows.cmd -VerifySafeUpdate`。验收必须确认已安装的受管 `Script.js` 与当前安装包及保存档位一致，并逐份确认 `proxy-groups` 是非空的块状或行内列表，再做 YAML、Mihomo 和自动更新状态检查；缺少代理组或列表为空时整批恢复，不能判成功。失败时只恢复 SHA-256 仍与检查时一致的文件，并发变化不覆盖且保留清单。仍须把运行内核写成“未验证”并重新检查，不能把文件恢复冒充运行状态恢复。客户端没有“全部更新”时逐份触发，但必须覆盖清单中的每一份并逐份报告。
5. 安全更新是用户主动触发的单次操作。档位 3 更新完成后继续保持自动更新关闭；档位 1、2 不替用户改自动更新设置。

共同基线用受管 `cn.mrs` 规则提供器同时驱动 DNS 与路由：`nameserver-policy` 的 `rule-set:<受管名称>` 把国内域名交给大陆加密 DNS，`RULE-SET,<受管名称>,DIRECT` 让连接直连。档位 3 另外保留 `geosite:cn` 后备，并让普通国外域名使用通过原主代理组访问的受管 DoH、AI 域名使用通过 AI 分组访问的受管 DoH。`direct-nameserver` 作为直连出站的同一套后备，不得为单个国内网站添加 DNS 例外。

订阅已有可选 AI 分组时，直接复用，只补全规则；不得修改它的成员或当前选择。没有 AI 分组时才创建独立的 `🤖 AI · Clash Patch`，加入订阅全部可用的真实节点和代理提供者，让主代理组与 AI 节点互不影响。订阅只有 AI 分组时允许普通与 AI 流量暂时共用它，不能因此跳过共同基线或档位 3 增强。找不到任何可用节点或代理提供者时不创建无意义的分组。不得创建第二个安全代理分组，也不得替用户选择节点。

规则列表最前面必须依次放置 `NETWORK,UDP,<AI 分组>`、`NETWORK,UDP,REJECT`。所有 UDP 都先由 AI 分组处理，不能回到本地直连；所选节点不支持 UDP 时继续命中拒绝兜底。AI 分组当前选择必须是代理节点，不能是 `DIRECT`；节点不支持 UDP 时如实报告 WebRTC 未通过或 UDP 不可用。

不要设计按应用隔离。用户会在同一个浏览器里同时访问国内网站和 AI 网站，而 WebRTC 的 STUN 连接不带原网页域名，Mihomo 无法判断它来自哪个标签页。TCP 与 DNS 可以按域名区分，WebRTC 防护必须覆盖全局 UDP。QUIC 也使用 UDP，因此会经过 AI 分组；国内网页的 TCP 和 DNS 仍然直连。明确告诉用户：这会影响游戏、语音和视频通话；家宽通常更适合 AI，最终节点仍由用户选择。

逐份报告结果。当前订阅只有自动刷新和运行检查都通过才能写成“已更新并自动生效”；失败并恢复时如实报告。其他订阅写成“选择该订阅时生效”。同时说明主代理组、AI 分组、是否新建了独立节点选择器和配置中的 TUN 状态。

## Patch 专用验收（Diagnostics 不固定执行）

macOS 运行 `ruby scripts/macos/verify_routes.rb`，Windows 运行 `powershell.exe -NoProfile -File scripts/windows/verify_routes.ps1`；Windows 默认从 Mihomo `/rules` 的最后一个 `MATCH` 读取当前实际主代理组，不能另用固定名称猜测。两者启动的 curl 都绑定独立源端口，只接受源端口、TCP、目标域名和新连接 ID 同时匹配的 Mihomo 实时记录，不能拿浏览器或后台程序的并发流量代替。访问 Google 时必须经过主代理组，或订阅明确提供且当前选择有效代理节点的非 AI 专用组，不能经过 `DIRECT` 或 AI 分组；访问 OpenAI、Anthropic 或 Claude 时必须经过 AI 分组当前节点。任何一项走错都要继续修复，不能只看 YAML 规则。

再验证用户报告的国内站点；没有指定站点时使用常见国内域名。通过 Mihomo `/dns/query` 读取当前 A/CNAME，同时直接查询策略中的两套大陆 DoH，确认当前结果来自同类大陆 CDN，而不是代理节点所在地的解析结果。随后记录 HTTPS 的 DNS、TCP、TLS、首字节和总耗时，并从 `/connections` 确认连接链为 `DIRECT`。解析位置、实时时间和连接链缺一项都不能写成通过。

然后使用 Computer Use 自动打开以下页面并完成操作。它们会收到浏览器公网 IP；用户已经要求运行本 skill 时可视为同意执行这些指定测试：

1. `https://ipinfo.cv/webrtc-check`
2. `https://ip.net.coffee/dns/`，点击“深度测试”
3. `https://ip.net.coffee/webrtc/`

三项都没有红色提示，而且没有未代理公网 IP、私网地址或本地运营商 DNS，才能说验证通过。macOS 或 Windows 环境只要提供 Computer Use，就由代理连续完成，不让用户代点。当前环境没有 Computer Use 时，给出中文逐步操作，写成 `未验证：需要手动测试`，要求用户在任何红色结果出现时截图发回，并继续修复直到通过。

不要修改或添加个人化的 Apple、iCloud 或测速网站规则。
