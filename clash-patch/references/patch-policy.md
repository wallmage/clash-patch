## 支持范围

支持：

- macOS 上使用 Mihomo 内核的最新版 ClashX Meta；
- Windows 上使用 Mihomo 内核的最新版 Clash Verge Rev。

要求 Mihomo 1.19.27 或更高版本。旧版 ClashX、旧版 Clash Verge、找不到内核、版本过旧或无法确认内核能力时只检查，不修改。macOS 建议升级到 ClashX Meta；Windows 建议升级到 Clash Verge Rev。

绝对不要退出、停止或重启 Clash 客户端。不得执行、建议或要求用户执行这类操作。中国用户通常依赖客户端越过 GFW；关闭客户端会让 AI 助手断线，并可能让修复停在一半。

## 模块选择

Clash Patch 有两个独立模块：

- **Patch** 负责首次安装或用户明确要求的完整安全增强。
- **Diagnostics** 负责从用户的模糊描述出发，主动取证、解释、单项修复并复测。

用户说“网络不对劲”时先进入 Diagnostics。不能因为用户提到 Clash、DNS、代理或某个网站，就直接运行安装程序。单项 Clash 配置问题也留在 Diagnostics；只有用户明确要求完整安全增强时才进入 Patch。

## Diagnostics 模块

### 工作目标

用户不需要会看日志、懂网络术语或先判断问题属于哪一层。代理负责选择工具、读取本机证据并推进调查；不要求用户先知道该查什么。任何外部记录都不是诊断前提，用户主动提供时才作为补充证据。

Diagnostics 默认只读。没有证据不能下结论，也不得直接执行 Patch 模块。先保留当前网络、客户端和应用状态；确认需要修改后，才做与证据对应的最小改动。

### 通用工作循环

1. **把描述变成可观察的症状。** 从用户原话中提取目标、动作、开始时间、频率和预期表现。能从当前窗口、应用、截图、系统状态或日志自行确定的信息直接读取，只在缺少登录、实体操作或不可推断的用户选择时提一个简短问题。
2. **记录修改前基线。** 始终记录时间、操作系统、活动网络和原始症状。有 Computer Use 且症状能在界面看到时，必须用用户实际使用的应用、账号、目标和动作在修改前复现；`curl`、配置检查或另一个浏览器只能作为分层证据，不能替代原场景。无法复现时保持“尚未证实”，不得写“已修复”。只有影响范围或证据指向共同网络路径时，才读取 Clash 客户端与 Mihomo 版本、当前模式、配置标识、TUN 状态和代理组选择。不得输出密钥、订阅地址或完整节点地址。
3. **复现并缩小影响范围。** 优先复现用户的原操作。用范围矩阵判断是单个目标、一个应用、当前设备、当前网络还是共同路径：至少包含异常目标的单变量开关对照，以及至少两个健康对照在可疑组件开启时的表现。只有异常目标受影响时，先查目标与组件的交互；健康对照也出现同类故障时，才检查、修复或替换共同组件本身。一个应用异常而其他应用正常时先查应用层；多个应用同时异常时，再向系统、Clash 和网络层推进。相同服务不等于相同账号、接口或认证方式，对照成立前必须确认真正保持不变的条件。
4. **建立假设表。** 每个候选解释都记录支持证据、反证、下一项最便宜的验证和状态。状态只能写成 **已确认**、**有力支持**、**尚未证实** 或 **已排除**。单个现象或相关性只能算线索；关键判断至少需要第二种独立证据，并且必须主动寻找反证。
5. **按需取证。** 按现象选择必要层级，不是每次全部执行。每完成一项检查，都更新假设的状态和优先级，保留所有尚未证实的解释，再选下一项最有区分度的检查。只有充分反证能排除某项解释时，才能标成“已排除”。
6. **检查解释是否完整。** 结论必须能解释全部已知现象，包括成功的对照、失败时间、范围和可量化差异。解释不了的观察必须保留，不能为了收尾而忽略。
7. **修复。** 一次只改变一个变量，保留修复前的状态和恢复方法。配置缺陷不等于故障原因；与原始症状无关但值得保留的增强必须单独说明，不能冒充修复。能在本机安全修复且属于用户要求的，由代理自动完成；需要管理员权限时由工具发起授权，不让用户抄命令。关闭应用、注销账号、删除或隔离缓存、Repair、重装或降级可能影响未保存工作、登录状态或本地数据，执行前必须说明精确对象、备份与恢复办法，并取得明确授权。涉及节点、代理组、订阅或 TUN 时也遵守本项目边界。
8. **针对原始症状复测。** 使用同一目标和动作比较修改前后结果，并检查健康对照没有变坏。间歇性问题要覆盖足够长的观察窗口；没有再次复现时写“暂未复现，观察中”，不能写“已修复”。

如果连续两次假设或修改没有改善原始症状，立即执行**诊断重置**：停止继续试改，恢复全部未生效的试验，重新列出已确认、已排除和未知项，并从原场景重新测量各层时间线。没有新的证据不得进行第三次修改。不得把前两次失败当成继续猜第三个设置的理由。

### 判断防线

| 容易误导用户的想法 | 必须执行的动作 |
| --- | --- |
| “命令行很快，页面应该已经好了” | 回到用户实际应用，用 Computer Use 验证原始动作。 |
| “这个配置确实不理想，所以就是它” | 把配置问题与当前故障分开；先做单变量对照和寻找反证。 |
| “关掉一个网站的过滤后变快，整个过滤器都坏了” | 完成范围矩阵；不能因为单个目标的对照结果就停用或删除整个组件。 |
| “又发现一个慢网站，再加一个例外” | 测试至少两个健康对照；同类故障跨目标出现时修共同组件本身。 |
| “前两次没用，再试一个设置” | 诊断重置；没有新的证据不得进行第三次修改。 |
| “这次打开挺快” | 对无副作用场景用同一应用、目标和动作连续测试三次。 |

如果现有记录覆盖不到异常时段，自动建立有开始时间、结束时间和采样间隔的观察窗口。必须记录采集工具或会话、进程或任务标识、采集范围、开始与结束时间、停止与清理方法，并确认采集正在运行。优先记录计数、时序和连接元数据；除非更轻量的证据不足且用户已授权，不抓取通信内容。问题再次出现后，按时间线关联证据并关闭观察，不让监测无限运行。工具无法持续运行时写“未建立监测”，不得承诺下一次会自动采集。

观察时长由原始故障决定。可以立即重复的问题，修复前后各连续测试三次。间歇性问题优先覆盖最近记录中的典型复发间隔；捕捉到一次异常后，修复后至少观察同样长的时间窗。没有历史可参考时先建立短窗口，仍未出现就按用户描述的频率延长，并保持“观察中”。

连续复测不得增加账号或业务风险。重复动作会提交登录、支付、发消息或其他有副作用的请求时，先检查失败次数、限流、验证码和锁定风险；出现迹象就停止重复提交，改用日志、只读接口或无副作用对照。Computer Use 可以读取界面并执行不含敏感输入的步骤；密码、验证码、MFA 或硬件确认由用户完成，代理不得记录或转述。

### 证据选择

以下是可选证据层，不是固定清单：

- **应用与浏览器：** 有 Computer Use 时必须用它重现操作，读取可见错误、加载状态和开发者工具中的失败资源。把 DNS、主文档与关键资源的网络耗时、首次显示和加载结束分开记录；主文档很快但页面仍空白时，优先检查脚本、Service Worker 和扩展注入，不得继续归咎 DNS 或 Clash。比较同一目标在另一个浏览器、无扩展窗口或同应用新会话中的表现；怀疑扩展或过滤器时先完成异常目标与至少两个健康对照的范围矩阵。单站对照无改善立即恢复；只有跨目标证据成立时才修共同组件本身。
- **DNS：** 比较系统解析、Mihomo `/dns/query`、A/CNAME、缓存状态和相关 DoH 的结果。只有证据指向解析或 CDN 错位时才清缓存或改 DNS；`DIRECT` 规则本身不能证明解析路径正确。
- **分流与代理：** 从 Mihomo `/connections` 读取目标、进程、命中规则、`rulePayload`、连接链、出站和字节计数。代理内核持有连接只说明流量经过它，不能单独证明是内核发起了业务请求。
- **传输：** 按需要测 DNS、TCP、TLS、首字节和总耗时，以及丢包、抖动、重传、接口错误和可用带宽。网页慢、连接失败和实时音视频卡顿使用不同指标，不能拿一次测速替代原场景。
- **系统与进程：** 查看当前进程、Socket、网卡计数、路由表和实时收发变化，再定位产生连接的应用。先看元数据和时间范围，避免无边界扫描用户文件。
- **历史与日志：** 先锁定异常时间线，再读取相关系统记录和应用日志。通过进程、连接目标、文件更新时间和事件类型定位日志，不按一张预设应用名单搜索。大日志、进程存在或时间接近都只是线索，必须与独立证据交叉验证。
- **泄漏与安全：** 只有用户要求安全检查，或现象指向 DNS、WebRTC、路由绕过时，才运行相应网页与连接验证。不得把完整补丁验收当成每次诊断的固定步骤。

### 重叠接管与职责分层

多个无关网站或应用都在内容开始传输前等待近似固定时长、随后快速完成时，先建立网络接管清单：系统代理或 PAC、透明代理或内容过滤器、VPN 或 TUN、DNS 过滤器、防火墙和终端安全软件。设置同时开启只能证明存在重叠，不能证明发生冲突；至少取得以下两类证据：原场景的等待与握手超时、重置或假地址处理失败在时间上对应，以及保持其他条件不变时切换一个组件的兼容模式能稳定消除等待。

确认重叠接管后做职责分层：同一流量层只保留一个透明接管者；内容过滤只覆盖浏览器时，可以把它放到遵守系统代理的应用层，让 VPN 或 TUN 继续负责最终分流、DNS 和 UDP；需要覆盖全部应用或特殊协议时，选择厂商支持的其他组合或明确排除范围。具体模式由用户需要的覆盖范围和厂商限制决定，不能把某个产品的兼容开关写成通用答案。

修复验收同时包含三类结果：原始症状在至少三个不同目标上连续复测；广告过滤、证书处理或应用覆盖等原有功能覆盖仍符合用户需要；分流、DNS、WebRTC 等安全属性保持不变。删除诊断期间添加的临时站点规则，不逐站添加例外。任何一类不通过都恢复修改前的接管模式，并保留“尚未解决”。

macOS 当前流量优先使用 `nettop` 的按进程/连接 CSV、活动监视器的网络计数、`netstat -ib`、系统 DNS/路由状态和 Mihomo 连接记录。macOS 不保证保存一份可直接按应用回放的长期流量历史；历史不足时建立观察窗口，并用相关应用日志补充，不能假装已取得过去的逐应用字节。

Windows 历史总量优先读取 `ConnectionProfile.GetNetworkUsageAsync` 返回的 `NetworkUsage`，按时间粒度汇总 `BytesSent` 与 `BytesReceived`，并覆盖异常期间使用过的全部网络配置；当前网卡使用 `Get-NetAdapterStatistics`。网页分阶段耗时优先使用系统自带的 `curl.exe`，TCP 端口对照使用 `Test-NetConnection`，连续时延和丢包使用 `Test-Connection`；实时应用优先读取应用自己的质量统计。需要捕捉复发问题时，可使用有明确时间和目标范围的 `pktmon` 计数或过滤采集。SRUM 或受保护的逐应用数据只有在确有必要、获得权限且能安全只读时才使用；聚合总量不能冒充逐应用归属。

日志与开发者工具只读取必要时间窗和必要字段。访问令牌、refresh token、ID token、`Authorization` 头、Cookie、客户端密钥、账号标识、会话内容、请求正文和完整本地路径不得写进诊断记录、聊天或搜索请求；能在本机完成的匹配只输出脱敏后的计数、时间、事件类型和结论。

### 判断与修复

禁止把“可能是”包装成结论。代理应优先设计能区分两个候选解释的检查，而不是继续罗列可能性。涉及数量时核对时间范围、单位、累计值与增量值，并检查推导是否能覆盖观察到的数量级；数学接近仍需独立证据确认对象和行为。

证据指向 Clash 配置时，先区分单项故障和完整安全增强。Patch 模块会应用完整安全策略，不是单项修复器。先只读比较拟议改动；如果包含与已确认问题无关的改动，用户又没有要求完整增强，就不得把完整 Patch 伪装成单项修复。

**macOS 单项配置事务：** 留在 Diagnostics，只修改证据确认的字段。保存原始字节和代理组选择；用 YAML 1.2 生成候选；重新读取候选；再次执行相同单项转换并要求无变化；用受支持的 Mihomo 在 30 秒内校验。写入当前订阅后只通过本地控制器刷新，并依次清除 Fake-IP 和 DNS 缓存，再复测原始症状。任何一步失败都恢复原始字节和原运行配置。不得调用完整安装器来代替这个事务。

**Windows 单项配置事务：** Windows 当前没有安全的即时单项配置写入路径。客户端运行时不得改当前配置；全局脚本又代表完整 Patch，不能用于伪装单项修复。完成诊断后报告已确认问题与生效边界；只有用户明确要求完整增强时才安装全局脚本，并等待客户端以后正常加载或刷新。

证据指向浏览器、应用、系统网络或本地缓存时，只修对应层，不顺手改 Clash。证据指向节点或服务端时，不擅自切换节点；说明已经排除的本机因素和用户需要做的最少选择。

Windows 客户端运行时不得修改当前配置；安装器只能更新全局脚本，等待客户端以后正常加载或刷新订阅。此时只能写“已更新，尚未生效”，不能写成已立即生效或“已解决”，也不得为了复测而触发订阅切换、节点切换、代理组切换或 TUN 切换，更不得重启客户端。当前会话只能完成只读验证，或修复不需要改运行配置的那一层，并如实说明生效边界。

本机无法修复，或修复依赖产品当前机制时，先确定问题类别，再在线搜索当前的官方或第一方资料；找不到第一方资料时才补充可靠的技术来源。搜索结果必须与本机证据吻合，并整理成用户可以直接照做的方案，不能只给链接或泛泛建议。

### 解释与完成标准

最终说明面向小白，按以下顺序写：

1. **发生了什么：** 一句话结论和证据状态。
2. **证据：** 两到四项最能区分其他解释的观察。
3. **为什么会这样：** 用日常语言解释，术语出现时立刻说明含义。
4. **做了什么：** 自动完成的修复，或本机无法修复时的明确方案。
5. **复测：** 原始症状的修改前后结果和观察时长。
6. **仍未确认：** 缺失证据和观察状态。只有已经确认采集器正在运行时，才说明下一次异常会记录什么；未建立监测时直接说明限制。

完成必须同时满足：结论有证据、修复针对已确认问题、原始症状已复测、没有引入新的异常。任一项缺失就报告实际状态，不写“已解决”。

## Patch 检查顺序

1. 识别操作系统和客户端。
2. 找到客户端配置目录。
3. 先读取客户端当前使用本地存储还是 iCloud，只选择当前存储位置；不得同时扫描两个位置。
4. macOS 只枚举当前存储位置中客户端实际使用的顶层 `.yaml`、`.yml`；排除旧 iCloud 容器、缓存、备份、日志和临时文件。`config.yaml` 是 ClashX Meta 的默认基础配置，不得删除；当前订阅不是 `config` 时安静跳过，只有它本身被选中时才处理。
5. 找到当前订阅和主代理组。
6. 执行平台安装程序。
7. 检查是否移除了旧版危险持久化机制。
8. 当前订阅修改后，通过 Mihomo 本地控制器自动刷新并完成运行检查；失败时恢复原文件和原运行配置。
9. 验证实时分流，再完成浏览器测试。

任何时候都不能输出整份配置。日志只能出现配置显示名称、处理状态、代理组名称和节点显示名称。

## DNS 与 TUN

共同策略：

```yaml
ipv6: false
tun:
  enable: true
  stack: system
  dns-hijack:
    - any:53
    - tcp://any:53
  auto-route: true
  auto-detect-interface: true
  strict-route: true
dns:
  enable: true
  ipv6: false
  respect-rules: true
  use-hosts: true
  use-system-hosts: true
```

补丁不创建安全代理组。`DNS` 出站只把请求交给 Mihomo 内部 DNS 模块，不是远端代理。Fake-IP 模式会在连接规则判定前解析域名，因此必须在 `nameserver-policy` 中把 `geosite:cn` 固定到大陆 IP DoH，并通过 `DIRECT` 连接；不能只设置 `direct-nameserver`。普通国外查询使用带原主代理组标签的受管 DoH，AI 域名查询使用带 AI 分组标签的同一套受管 DoH。这样国内直连网站获得大陆 CDN，AI 的解析请求和网页连接使用同一个 AI 出口。

两组 DoH 都直接连接解析器 IP，无需先解析解析器域名，避免解析器域名错误引发证书失败。代理侧使用 AdGuard 的两个非过滤地址和 TWNIC Quad 101；直连侧使用阿里 `223.5.5.5` 和 DNSPod `1.12.12.12`。所有查询都使用 HTTPS，不加入广告拦截，不发送 ECS。升级旧版补丁时，删除能确认属于本工具的 `🛡 安全代理 · Clash Patch`；普通代理 DNS 迁移到原主代理组，AI 域名 DNS 和 UDP 迁移到 AI 分组。

`default-nameserver` 和 `proxy-server-nameserver` 属于网络启动边界，必须保留用户值。`proxy-server-nameserver` 缺失时使用 `system`，满足 Mihomo 的 `respect-rules` 要求；如果启动字段等于旧版补丁写入的固定 `1.1.1.1`、`8.8.8.8` 组合，也迁移为 `system`。不得把可直接访问的节点域名解析器替换为可能被阻断的境外地址。

`direct-nameserver` 不保留原值，统一写成策略文件中的大陆 IP DoH；同时把 `direct-nameserver-follow-policy` 设为 `false`。`nameserver-policy` 中的 `geosite:cn` 也必须覆盖为同一组解析器，避免 Fake-IP 初次解析先落到代理侧。这样不会让用户原有的 `system`、明文 DNS 或代理 DNS 使国内域名继续泄露或获得境外 CDN。直连 DoH 会让阿里或 DNSPod 看到国内域名查询，但本地运营商只能看到加密的 HTTPS 连接；这属于受管的分流，不是意外泄露。

如 `nameserver-policy` 把多个域名写在同一个逗号分隔键中，拆成独立键。只有当解析器片段指向已知的非直连节点，或指向依赖关系中无法到达 `DIRECT` 的静态代理组时，才保留这个分流目标；解析器地址统一换成策略文件中的三个 IP DoH。这样不会因为 `dns.google`、`cloudflare-dns.com` 等解析器域名被错误解析而让所有新域名一起失败，也不会继续依赖某些节点会拒绝的 `8.8.8.8`、`1.1.1.1`。判断代理组时必须应用 `exclude-filter`；筛选后为空时，只接受明确指向安全内联代理的 `empty-fallback`，默认的 `COMPATIBLE` 不合格。带 `use`、`include-all` 或其他动态成员的组无法静态证明，改为受管 DoH。Mihomo 的 `exclude-type` 只作用于自动纳入的节点，这些动态组本来就不保留。

`#h3=true` 可以保留。除受管的 `direct-nameserver` 外，`#en0`、`#RULES`、未知名称、明文 DNS、`DIRECT`、`DNS` 出站和没有代理组标签的查询都改为代理侧受管 DoH。带 `skip-cert-verify=true` 的解析器跳过证书校验；带 `ecs` 或 `ecs-override` 的解析器会发送客户端子网信息。这三类参数也改为受管 DoH。

不要为某个打不开的网站添加专用 DNS 例外，也不要为测速成绩添加国内 DNS 例外。国内 CDN 错位时检查 `direct-nameserver` 和实时出站，不添加站点规则。同类网站一起出现 `SERVFAIL` 时，应检查整组解析器和当前节点；某个配置在另一节点上恢复，通常是节点故障，不要误改 DNS。最后以真实泄漏测试结果为准。

## 主代理组

按以下顺序选择。AI 命名的组在任何一步都不作为主组候选，避免 AI 组把流量指向自己：

1. 最后一个 `MATCH` 指向的非 `DIRECT` 可选代理组；
2. 被宽泛规则（`MATCH`、`GEOIP`、`GEOSITE` 或广域 `RULE-SET`）引用超过一次的可选代理组；
3. `Proxy`、`PROXY`、`Final`、`Fallback`、`节点选择`、`节点列表` 或 `兜底分流`；
4. 第一个非纯 `DIRECT` 的 `select` 组。

找不到主代理组时不要猜单个节点。保持原文件不变，并告诉用户需要一个可选主代理组。

## AI 组

先寻找订阅已有的可选 AI 分组。名称匹配策略文件中的 AI 组名称，或包含独立的 `AI`、`OpenAI`、`人工智能` 时，视为已有 AI 分组。已有时直接复用，只补全规则；分组类型、成员、顺序、图标和当前选择全部保持原样。

只有确认没有可选 AI 分组时，才创建 `🤖 AI · Clash Patch`。新组直接列出订阅中全部有效的内联代理节点，并通过 `use` 接入全部有效的 `proxy-providers`。不得只把主代理组或“节点列表”作为唯一成员；否则用户无法让普通流量与 AI 流量选择不同节点。名称被其他代理组或内联代理占用时使用没有冲突的编号。重复运行必须复用已创建的组，不能产生重名。没有任何有效内联节点或代理提供者时不创建分组，保持原文件不变并说明原因。

升级旧版补丁时：订阅已有自己的 AI 分组，就删除能确认属于旧补丁的重复 AI 组并把补丁规则改到原有 AI 分组；订阅没有其他 AI 分组，就保留旧补丁 AI 组并把它迁移为上述独立节点选择器。

AI 组负责 OpenAI、ChatGPT、Codex、Claude、Anthropic，以及策略文件中列出的 Google AI 和相关服务流量。

## 节点建议

用户原有 AI 分组的成员保持不变。只有 Clash Patch 自己新建或旧版创建的 AI 分组会填入全部可用节点和代理提供者。补丁不得调用控制器切换节点，也不得替用户选择台湾、日本或任何家宽节点。

AI 分组的用途是分开普通流量和 AI 流量：主代理组可以选择低倍率节点用于浏览、下载和视频，AI 分组可以由用户单独选择信誉更好的家宽节点。补丁只提供完整选项，不替用户决定。

聊天中说明家宽通常比数据中心节点更适合 AI 服务；如果用户有台湾家宽，优先建议台湾，没有台湾时可建议日本。建议只写在聊天中，最终选择完全交给用户。

## AI 规则

唯一数据来源是 [policy.json](policy.json)。Windows 脚本中的策略块由本地生成器写入，并由一致性测试检查，不能手工维护另一份规则。

规则覆盖：

- `anthropic.com`、Claude 应用、内容、MCP 和静态服务；
- `openai.com`、`chatgpt.com`、Codex 所使用的 OpenAI 域名；
- OpenAI 实时语音、静态资源、上传和明确属于 OpenAI 的验证服务；
- 已有补丁中的 Google AI 和 Gemini 域名；
- Anthropic 官方公布的入站 IPv4 `160.79.104.0/23` 与 IPv6 `2607:6bc0::/48`，并使用 `no-resolve`；
- `DOMAIN-KEYWORD,openai` 后备规则。

明确禁止把以下通用域名整体交给 AI 组：

- `raw.githubusercontent.com`
- `storage.googleapis.com`

不得把整站 `sentry.io`、`auth0.com`、`segment.io`、`intercom.io`、Stripe、Cloudflare Challenge、SendGrid、WorkOS 等共享服务交给 AI 组。只保留第一方域名和能确认专门服务于 AI 产品的精确主机名。
`ai.com` 属于无关产品，也不得加入 AI 规则。

从旧策略升级时，把目标仍是补丁自有 AI 组的 `DOMAIN-SUFFIX,ai.com` 和 `IP-CIDR,160.79.104.0/21` 当作迁移项删除，并写入当前的 `/23`。相同旧规则若指向用户组，必须保留；DNS 中旧的 `+.ai.com` 也只在解析器仍指向补丁自有安全组时删除。

只删除目标为受管 AI 组的上述规则。其他用途的规则保持不变。

订阅把同一批 AI 域名交给主代理组的通用规则必须替换，否则会让 AI 流量绕过独立 AI 分组。用户明确写给其他非主代理目标的同类规则保持原目标，并排在补丁规则之前。AI 明确规则必须出现在国内直连、GEO、所有 `RULE-SET` 和 `MATCH` 之前。

## WebRTC

依次加入两条规则：

```text
NETWORK,UDP,<AI 分组>
NETWORK,UDP,REJECT
```

第一条的目标使用 AI 分组，不创建安全代理分组。把两条都放在规则列表开头，确保更靠后的 `NETWORK,udp,DIRECT`、`DST-PORT,3478,DIRECT`、进程规则和任意 `RULE-SET` 都不能先放行 UDP。已有规则不删除；除了与补丁 AI 规则同键的用户规则会前移以保留优先权，其余规则保持原目标和相对顺序。

AI 分组当前选择必须是代理节点，不能是 `DIRECT`。节点不支持 UDP 时 Mihomo 继续匹配拒绝兜底，不得写成 WebRTC 已验证；它不会因此改走本地直连。补丁不得切换分组或节点，只能检查并如实报告当前状态。

不能按浏览器应用或 AI 域名单独防 WebRTC。同一个浏览器会同时访问国内和 AI 网站，而 STUN 连接的目标与原网页域名分离，Mihomo 无法判断连接来自哪个标签页。TCP 和 DNS 按域名分流；WebRTC 防护必须覆盖所有 UDP。HTTP/3/QUIC 也经过 AI 分组；国内网页的 TCP 与 DNS 仍然直连。在线游戏、语音和视频通话也可能受影响，安装前后的说明都必须写明这个影响。

AI 网页流量、AI 域名 DNS 和所有 UDP 使用同一 AI 分组。多个结果如果全是代理出口，不属于真实 IP 泄漏；单一结果更容易确认。

## 订阅刷新

### macOS

macOS 只允许单次显式运行。不得安装 LaunchAgent、`RunAtLoad`、`WatchPaths`、计划任务或目录监听。安装程序必须移除能通过 `Label` 和程序参数确认所有权的旧版 `com.clashpatch.profiles` 与 `com.wallny.clash-profile-patcher` LaunchAgent；无法确认所有权时保持不动并报告。

只处理 ClashX Meta 当前存储位置。使用本地存储时只看本地目录；使用 iCloud 时只选包含当前订阅的当前容器。读不到存储模式，或多个 iCloud 容器无法通过当前订阅唯一判断时停止，不猜。旧容器、历史目录和备份即使仍存在也不得处理。

订阅使用 YAML 1.2 核心类型读取，`yes`、`on`、`0123`、`1:20` 和日期保持文本。多文档 YAML 与循环别名按无效内容跳过；超过系统 Ruby 安全递归深度的文件只跳过该文件，不能终止整批。写入候选文件前先重新解析，并强制使用满足最低版本的 Mihomo 检查。读取、校验和写入之间若订阅被刷新，重新读取并重试，连续变化时跳过。最终写入必须通过已锁定的原文件描述符完成；若客户端原子替换路径，旧描述符不能覆盖新订阅。首次备份使用独占创建，两个安装进程也不能互相覆盖备份。符号链接订阅写入真实目标，不能用重命名替换符号链接本身。

候选 YAML 重新读取后，必须再次执行完整转换。二次转换仍报告变化，或二次结果与第一次候选不同，说明补丁不具备幂等性；此时保留原文件并写成“已跳过：二次转换不一致”。

当前订阅写入前记录当前所有可选代理组的选择。写入后只允许通过 Mihomo 本地控制器执行 `PUT /configs?force=true`，并要求 HTTP 204。刷新成功后依次执行 `POST /cache/fakeip/flush` 和 `POST /cache/dns/flush`，避免旧 Fake-IP 映射继续使用代理节点所在地的 CDN；任一请求失败都视为刷新失败。

缓存清理后确认 TUN 仍启用、原有代理组选择没有变化、国内和境外 DNS 查询均成功、外网可连接。对用户报告的国内站点，还要比较 Mihomo 与两套大陆 DoH 的 A/CNAME、记录 HTTPS 分阶段耗时，并确认实时连接为 `DIRECT`；解析仍随代理节点位置落到境外 CDN 时继续修复。任何一步失败，都必须在当前进程内恢复写入前的原始字节，并在候选已被内核采用时再次加载原配置。控制器不可用时也恢复，不把修改后的当前订阅留给用户手动处理。不得调用 AppleScript，不得切换 TUN、订阅、代理组或节点，更不得退出、停止或重启客户端。

### Windows

Clash Verge Rev 每次加载或刷新订阅都会运行 `profiles/Script.js`。全局扩展脚本对传入配置应用相同策略。TUN 和顶层 IPv6 使用 Clash Verge Rev 的应用配置，因此不会被脚本后的应用设置覆盖。

Windows 不安装计划任务或后台服务。Clash Verge Rev 运行时只允许更新 `profiles/Script.js`，不得修改 `verge.yaml`、`config.yaml` 或当前运行配置。客户端原本没有运行时，才可用可信路径中的 Mihomo 做版本与候选配置检查，并按 YAML 映射节点修改应用设置。安装程序处理 `tun: null`、空格键名、行尾注释和块状 `dns-hijack`；无法安全合并的锚点与多文档 YAML 直接拒绝，`--- # 注释` 也算文档标记。全部候选先生成并检查，再写入；中途失败时继续恢复每一个目标，并按原始字节保留编码和 BOM。Windows 备份停止继承目录权限，只允许当前用户、SYSTEM 和 Administrators 完全访问。

Windows 全局脚本也必须做二次转换一致性检查；结果不一致时返回原配置。任何流程都不得结束、暂停或重启 Clash 进程，也不得要求用户这样做。

组合 `Script.js` 时，只把代码状态中的独立行注释当作补丁边界；字符串、模板字面量和块注释中的相同文字不是边界。已有脚本使用补丁保留标识符、异步 `main` 或会在重命名后误调用补丁入口的递归 `main` 时，安装器必须拒绝并保持原文件。

首次安装把 `config.yaml` 与 `verge.yaml` 的原始字节和安装后哈希写进本地状态文件。安装和卸载都必须校验版本、两份记录、布尔值、规范 Base64 和 SHA-256；不完整状态不能当成空状态。卸载时只有当前哈希仍与安装程序最后写入的版本一致才恢复；检测到安装后新改动时保留当前文件和状态，不能覆盖。Windows 的订阅增强只在各订阅随后加载或刷新时发生，安装完成不能说所有订阅已经处理。

## 异常处理

- `401 unauthorized`、HTML、空文件或损坏 YAML：写成“已跳过：订阅内容无效”，等待以后刷新出有效订阅。
- Mihomo 拒绝候选文件：保留原文件，写成“已跳过：内核校验失败”。
- Mihomo 版本检查或候选校验超过 30 秒：终止本工具启动的 Mihomo 校验子进程，保留原文件，写成“已跳过：订阅响应超时”，继续处理下一份订阅。
- YAML 候选的二次转换与第一次不同：保留原文件，写成“已跳过：二次转换不一致”。
- 找不到 Mihomo、版本低于 1.19.27 或无法读取版本：停止安装，不写入任何目标文件。
- 读取或写入失败：保留能保留的原文件，写成“已跳过：读取或写入失败”，不能伪装成订阅无效。
- 找不到主代理组：不修改，不猜节点。
- 没有已有 AI 分组，并且订阅中找不到任何有效内联节点或代理提供者：不创建空组，写成“未修改：找不到可用的 AI 节点”。
- 已有 Windows 全局脚本只有一个同步 `main`：先运行原脚本，再运行 Clash 补丁。Clash Verge Rev 当前不会等待 Promise，因此必须拒绝 `async function main`。
- 已有 Windows 脚本结构无法安全组合：保留原文件，请用户发送提示和 `Script.js` 截图。
- macOS 的 REALITY `short-id`：只保护已有且格式有效的十六进制文本；不补齐、不截断、不猜缺失值。Windows 脚本不改这个字段。
- 当前配置只有自动刷新和运行检查全部通过，才能写成“已更新并自动生效”。刷新失败并恢复时写成“自动刷新失败，已恢复原配置”；文件恢复但内核未恢复时必须单独说明。
- ClashX Meta 正在运行时保持运行，不调用 AppleScript，不修改 `restoreTunProxy`，不切换 TUN 或代理节点。
- 远程 `proxy-providers` 在 Mihomo 写入前检查中可能需要网络；离线检查失败时保留原文件，并提示联网后重试。

## Patch 验证标准

先生成 Google、OpenAI 和 Claude/Anthropic 的真实连接，同时读取 Mihomo `/connections`。Google 的连接链必须包含当前主代理组及其所选节点；AI 网站的连接链必须包含 AI 分组及其所选节点。只看配置文件或网页出口 IP 不足以证明分流正确。

然后必须测试：

1. `https://ipinfo.cv/webrtc-check`
2. `https://ip.net.coffee/dns/` 的“深度测试”
3. `https://ip.net.coffee/webrtc/`

三项都没有红色提示，并且不显示用户未代理的公网 IP、私网地址或本地运营商 DNS，才算通过。

三项均为第三方页面，会接收浏览器的公网 IP，并发起 WebRTC 或 DNS 测试请求。补丁不向这些页面上传订阅、节点密码或配置文件；用户可以拒绝使用，结果写成“未验证”。

macOS 和 Windows 只要当前代理工具提供 Computer Use，就由代理打开页面、点击“深度测试”并读取结果，不把刷新或测试停给用户。当前环境没有 Computer Use 时，要求用户手动测试。只要有红色结果，就让用户发截图并继续处理，不得说已经验证。
