# Clash Patch Skill Design

**首次确定：** 2026-07-20
**当前版本：** 2026-07-22

本文档固定记录产品需求和架构。详细配置规则保存在 `clash-patch/references/patch-policy.md`，开发流程保存在 `AGENTS.md`。

## 产品目标

Clash Patch 面向不懂网络配置的用户，包含两个模块：

- **Patch** 首次运行时询问并保存用户用途，只应用该用途需要的最少设置；完整 DNS、AI 分流和 UDP/WebRTC 防护只属于第三档。
- **Diagnostics** 接受用户模糊的网络故障描述，主动复现、取证、分析、解释、修复并复测，不要求用户先判断问题类型或指出日志位置。

用途档位同时约束 Patch 和 Diagnostics，是用户长期保存的网络需求，不是一次性安装参数。

失败时必须保留原配置并给出中文说明。真实案例只用于检验工作流，不作为固定问题、固定工具或固定答案写进产品。

产品不支持旧版 ClashX、旧版 Clash Verge 或 Linux，不包含 Apple、iCloud、Speedtest 等个人规则。

## 核心要求

- 绝不退出、停止或重启 Clash。
- 只有用户选择的档位明确要求时，才通过客户端界面切换 TUN 或 Clash 自己的系统代理；AdGuard for Mac 只允许按下述已知兼容路径在自己的界面中切换模式，不改写第三方 PAC，也不切换订阅、代理组或节点。
- 只处理客户端当前存储位置，不扫描旧容器、备份或废弃订阅。
- 保留 ClashX Meta 的默认 `config.yaml`；当前未选中它时安静跳过。
- 已有 AI 分组只补规则并保留原有成员；没有时创建包含全部真实节点和代理提供者的独立 AI 选择器。
- 主代理组与 AI 节点独立选择，让普通流量可走便宜节点、AI 流量可单独走家宽；不创建安全代理分组，也不自动替用户选择节点。
- 三个档位用同一受管 `cn.mrs` 规则提供器同时驱动 `nameserver-policy` 与 `RULE-SET,...,DIRECT`，覆盖当前存储位置中的全部订阅；档位 3 另让普通国外域名使用带原主代理组标签的 IP DoH、AI 域名使用带 AI 分组标签的 IP DoH。保留节点启动解析。
- 所有 UDP 先经过 AI 分组，节点不支持时拒绝兜底。HTTP/3/QUIC 因此不会直连；国内网页的 TCP 和 DNS 仍按域名直连。这是共享浏览器中避免 WebRTC 直连所需的取舍，不能把浏览器白屏自动归因于 QUIC。
- 不做按应用隔离：浏览器同时承载国内和 AI 网站，STUN 连接也不携带原网页域名，Mihomo 无法判断它来自哪个标签页。TCP 和 DNS 按域名分流，WebRTC 防护覆盖全局 UDP。
- 修改前备份，候选必须可解析、可通过 Mihomo 校验且二次转换一致。
- 不输出订阅地址、认证信息、控制器密钥或完整节点地址。

## 三档 Patch 架构

第一次在一台电脑上进入 Patch 时，必须问“你使用网络代理主要用于哪些用途？”，解释三个档位后保存数字选择。macOS 使用 `usage-profile.plist`，Windows 使用 `clash-patch-usage-profile.json`。用户可以修改；升档补充能力。从第三档降到第一、二档时先运行安全卸载，再保存新档位；只撤销能证明由本工具拥有且未被继续修改的设置，不能覆盖后来的用户改动，无法可靠恢复的旧订阅增强必须明确报告。

安全卸载成功后清除旧用途档位，失败时保留；这样档位 3 才能按“先卸载、再保存新档位”完成降档，同时不能绕过未完成的恢复。

1. **档位 1｜普通浏览**：给当前存储位置中的全部订阅安装共同国内域名直连基线，并确认 Clash 的“设置为系统代理”已开启；不修改 TUN、IPv6、WebRTC、AI 分组、节点或自动更新。用国内站、Google、Twitter 和用户常用站点验收。
2. **档位 2｜海外 AI**：继承共同国内域名直连基线；用于 ChatGPT、Codex、Gemini、Perplexity 等，不含 Claude。开启 TUN，关闭 Clash 自己的系统代理开关；按下述规则处理 AdGuard for Mac，不改写第三方 PAC，也不增加 WebRTC 或 AI 分组补丁。验收增加 ChatGPT、Gemini 和 Agent 联网。
3. **档位 3｜Claude/Claude Code**：先完成第二档，再运行现有完整补丁和完整验收。说明全局 UDP 对 QUIC、游戏、语音和视频的影响；只建议台湾家宽优先、其次日本家宽，不自动切换节点。

关闭 Clash 系统代理是为了避免与 TUN 重复接管，不是隐藏代理。明确的 Claude 或 Claude Code 网络配置需求可直接选择或升级第三档。三个档位都进入共同国内域名直连转换；第一、二档在完成共同基线后结束，客户端开关由 Computer Use 完成。只有第三档进入其余完整增强。

## Diagnostics 架构

Patch 和 Diagnostics 是两条独立工作流，但共享同一个用途档位。每次诊断先读取档位，并把它转换为需要保持的能力、禁止检查或修改的内容、修复后的回归项目。用户报告故障不能自动升档；只有用户明确增加用途时才改档。单项 Clash 配置修复仍属于 Diagnostics，并复用 Patch 的安全事务；完整增强只用于档位 3。

- **档位 1：** 维护全部订阅共同的国内域名直连基线，并保障普通境外网站与系统代理；不检查或修改 TUN，不运行 DNS 泄漏、WebRTC、AI 分组或 Claude 检查。
- **档位 2：** 继承共同国内域名直连基线，增加 TUN、普通海外 AI 与 Agent 联网；不运行 DNS 泄漏、WebRTC 或 AI 分组检查。
- **档位 3：** 同时维护普通浏览、海外 AI、Claude、DNS、WebRTC 和 AI 分流。共享网络路径发生修改时，复测本次改动可能影响的第三档能力；只改浏览器或单个应用时不机械运行无关测试。

诊断采用同一个闭环处理不同问题：

1. 把用户描述转成可以复现或监测的现象。
2. 有 Computer Use 且故障能在界面看到时，先在用户实际应用中复现；保存修改前基线，并用异常目标加至少两个健康对照建立范围矩阵。
3. 同时保留多个解释，按区分能力选择浏览器、应用、DNS、Mihomo 连接、传输质量、系统计数和相关日志证据。
4. 用独立证据和反证更新判断；资料不足时建立有期限的观察窗口。
5. 执行最小、可恢复的修复。两次假设或修改无效后恢复全部试验并做诊断重置，没有新证据不进行第三次修改。本机无法处理时，基于已确认的问题类别查询当前官方方案。
6. 在同一应用、账号、目标和动作中连续复测原始症状，并用小白能懂的语言报告结论、证据、修复、结果和未知项。

预计超过 10 分钟且能拆出多路独立只读证据时，主代理可以安排多个 Sub Agent 并行调查；它先固定档位、范围和统一时间窗，并保证同时只有一个界面操作者、一个主动流量生成者。Sub Agent 不写配置、不更新订阅、不恢复备份，也不作最终判断；写入、更新、恢复和结论由主代理串行完成。

Skill 不维护一张每次全跑的诊断清单。具体工具同时由现场证据和保存档位决定；完整 DNS/WebRTC 与 AI 分流验收只属于档位 3。macOS 使用实时进程、连接和接口数据，不能虚构不存在的历史；Windows 可以读取系统保存的网络用量，但聚合总量不能冒充逐应用归属。

浏览器诊断必须把主文档网络耗时、关键资源、首次显示和加载结束分开。配置缺陷不能直接当成故障原因。怀疑扩展、过滤器或共同网络组件时，目标开关对照只回答该目标是否受影响；还要用至少两个健康对照判断范围。只有单站异常时修目标交互；跨目标出现同类故障时修共同组件本身，不按网站不断加例外。重叠接管与职责分层也受档位限制：第一档不盘点或切换 TUN；第二、三档只有证据指向共同路径时才检查。修复后验证原始症状和当前档位中可能受影响的能力；无改善或功能退化立即恢复，不能因为浏览器问题顺手应用更高档位的 Clash 配置。

AdGuard for Mac 是 Patch 和 Diagnostics 共享的已知兼容路径。档位 2、3 检测到它以 `Network Extension` 与 Clash TUN 同时透明接管时，通过 Computer Use 在 AdGuard 自己的界面切换为“自动代理”，让 Clash TUN 负责最终分流、DNS 和 UDP，AdGuard 负责遵守系统代理的过滤。修改后复测至少三个无关目标、Safari 和 Chrome 的广告过滤，以及当前档位可能受影响的能力。Clash TUN 存在时不得把 AdGuard 改为 `Network Extension`；禁止按应用调整 AdGuard 过滤范围，因为产品不能穷举用户安装的应用。检测到应用本身不是故障结论；Diagnostics 通过单变量前后对照确认，无改善立即恢复。第一档需要 Clash 占用系统代理，Windows 的产品机制也不同，两者都不照搬这个组合。不得逐站加例外、改写 PAC、直接编辑 AdGuard 设置或停用、退出、卸载 AdGuard。

国内请求误走海外是第一档共享故障，档位 1、2、3 都处理。目标在国内数据集、实时连接却命中海外代理、而大陆 DoH 与直连稳定时，直接修复全部订阅的共同提供器、DNS 引用和路由引用，不为 Kimi、欧陆词典等单站加例外。非浏览器应用的证书身份错误单独归类：先核对系统代理所有权，把 CFNetwork 的 PAC 查询中断、回环端口所有者、TLS 失败和 Clash `ProxyConfigHelper` 活动放进同一时间线。档位 2、3 只允许“Clash TUN 开、Clash 系统代理关、AdGuard 自动代理开”；已经符合时不修改，改为建立观察窗口并重复原动作。绝不点击忽略证书、添加 Apple/iCloud 规则、关闭全局 HTTPS 过滤或归咎于订阅节点。

## 平台架构

两端按“用户可获得的能力”尽量保持对等，允许客户端机制不同：都支持用途档位、Diagnostics、配置历史、安全更新、档位 3 策略、实时分流和浏览器验收。Windows 客户端运行时的普通备份恢复只能比较和准备，不能停止客户端或把文件恢复假装成运行内核恢复；这是明确的客户端安全边界。Windows Computer Use 已在 2026-07-09 的桌面版中提供；当前会话实际启用时，Windows 与 macOS 都用它完成客户端开关、原应用复现和浏览器测试。Windows 只能操作前台桌面，设备需保持解锁；没有工具时不能声称已自动验收。脚本和本地控制器能可靠完成的步骤不改走界面。

### macOS

三个档位都检查 Mihomo，并逐份处理当前使用的本地目录或 iCloud 容器；第一、二档只写共同国内域名直连基线。第三档额外关闭自动更新：把 ClashX Meta 的 `kAutoUpdateEnable` 设为 `0`，保存修改前状态并回读确认，再应用完整增强。不得安装永久监听、LaunchAgent、`WatchPaths` 或计划任务。当前订阅写入后自动刷新并清除 Fake-IP 与 DNS 缓存；失败立即恢复。

安全更新是独立的显式事务。它从客户端清单读取全部远程订阅，先为全部当前文件建立带日期时间的备份，再下载、解析、按档位转换并校验全部候选。只有全部成功才统一替换；任一份失败时尝试恢复全部目标，确认所有目标都恢复后才能说“全部保持原样”，恢复本身失败时继续处理其余目标并明确报告。三个档位都对全部候选重打共同国内域名直连基线，档位 3 再打完整补丁。项目不保留订阅后台监听，档位 3 的自动更新保持关闭。

配置历史也是 Diagnostics 的证据。第一次运行保存初始快照，每次写入前保存带日期时间的修改前版本；出现网络故障时先列出备份，选择症状出现前的候选，只输出配置差异中的变化字段与哈希。时间只用于筛选候选，不能代替故障证据。回滚前再次备份当前版本，校验备份、目标与当前哈希；恢复后复测原始症状，恢复或验收失败时恢复回滚前版本。macOS 恢复当前订阅后必须通过本地控制器重新加载，保持恢复前的 TUN 与代理组选择，并完成缓存、DNS 和连接检查；文件恢复但运行内核未恢复时单独报告，非当前订阅不触发切换。Windows 客户端运行时不停止客户端来强行恢复。

Ruby 引擎使用 YAML 1.2，保护有效的 REALITY `short-id` 文本。每份候选重新解析、再次转换，并由 Mihomo 1.19.27 以上版本校验；校验最长 30 秒。文件通过锁、一次性私有备份和原文件描述符写入，避免覆盖同时刷新的订阅。

### Windows

三个档位都安装带数字档位的 `profiles/Script.js`，让订阅加载或刷新时应用共同国内域名直连基线；第一、二档不执行其他增强。第三档额外把 `profiles.yaml` 中每个远程订阅的 `option.allow_auto_update` 改为 `false`，修改前备份并逐项回读，不依赖 Computer Use；不安装额外后台服务。

客户端运行时只更新全局脚本和订阅清单中的自动更新字段，不碰当前运行配置；客户端未运行时，PowerShell 安装程序可以事务式更新应用级 TUN 与 IPv6 设置。已有同步 `main` 可以安全组合；结构不明确、异步或递归入口保持不动。

安全更新前从 `profiles.yaml` 核对全部远程 uid 与本地文件，建立脱敏清单并逐份备份；Computer Use 触发“全部更新”后，PowerShell 逐份验证 YAML、Mihomo 和自动更新状态。失败时在独占文件期间复核检查开始时的 SHA-256，只恢复没有再次变化的文件；并发刷新产生的新内容不会被旧备份覆盖，清单会保留并明确报告。Windows 使用自己的 `verify_routes.ps1` 从 `/proxies` 与 `/connections` 验证主代理组和 AI 分组的当前节点，与 macOS 验收条件一致。

## 公开接口与代码组织

所有公开命令都提供 JSON v1：macOS 使用 `--json`，Windows 使用 `-Json`。默认中文输出保持兼容；JSON 模式的标准输出只有一个对象，`exit_code` 与实际退出码一致，`code` 和 `operation` 可供自动化稳定判断。`command` 只取 `install`、`uninstall`、`patch`、`verify_routes`，字段和类型由 `clash-patch/references/result-contract.json` 定义。任何机器输出都不能包含 URL、密钥、完整本机路径或节点名称。Skill 调用脚本时优先读取机器输出。

公开入口、参数和调用方式不因内部调整而改变。macOS 与 Windows 的大脚本按配置转换、备份与事务、Mihomo 校验、订阅处理、安全更新、运行状态和 CLI 拆分；入口只做兼容转发与编排。模块边界不能改变先校验后写入、失败恢复、并发保护和客户端运行边界。

## 生效与验收

当前订阅只有本地控制器自动刷新和运行检查都通过，才能说补丁已经生效。随后先用 Mihomo 实时连接确认国内网站获得大陆 CDN 并走 `DIRECT`、Google 走主代理组或订阅明确提供的非 AI 专用代理组、OpenAI/Anthropic/Claude 走 AI 节点，再检查两项 WebRTC 页面和 DNS 深度测试。Google 不能走 `DIRECT` 或 AI 分组。有 Computer Use 时由代理完成网页操作；没有时给用户中文步骤并要求返回失败截图。只有分流正确、三项网页都没有红色提示，且没有未代理公网 IP、私网地址或本地运营商 DNS，才能说验证通过。

## 测试策略

`tests/fixtures/main_group_cases.json` 是 macOS Ruby 与 Windows JavaScript 共用的配置转换 conformance 案例。每个案例用人工确认后的 `expected_config_sha256` 锁定完整结果，两端必须读取同一输入、分别匹配该摘要，再对完整输出做深比较；每个发生变化的案例都要再执行一次，并确认第二次不再变化。

覆盖率随拆分后的全部生产模块统计：macOS 纯配置转换核心、独立 Ruby 分流验证器与 Windows JavaScript 转换引擎要求 100% 行覆盖，JavaScript 同时要求 100% 函数覆盖；包含操作系统适配、文件事务和 CLI 的 macOS 生产模块合计行覆盖不得低于既有门槛。代码移到新模块不能让统计范围缩小，门槛只能上调。PowerShell、Shell 与 CMD 以真实参数、退出状态、文件副作用和禁止副作用的行为测试为准，不用源码字符串或无断言执行凑百分比。备份先于写入、版本与路径门槛、二次转换、并发变化、自动更新、整批恢复、SHA-256 恢复保护、运行/未运行边界、机器输出协议、脱敏和超时都必须有回归场景。CI 同时禁止在生产脚本加入覆盖率忽略标记。

需要使用本机安装 Mihomo 的测试属于显式集成测试，不计入可重复覆盖率；默认测试使用受控内核模拟验证相同的命令、超时和失败状态。本机集成测试失败必须单独报告，不能通过降低断言或伪造通过来处理。

## 固定文档

- `README.md`：面向用户的完整功能说明：安装、用途档位、补丁内容、诊断方式、平台机制、安全更新、恢复与限制。
- `clash-patch/SKILL.md`：代理必须执行的流程。
- `clash-patch/references/patch-policy.md`：详细产品规则、异常和状态。
- 本文档：产品目标与架构。
- `AGENTS.md`：开发、测试和发布规则。
- `tests/baseline.md`：现行自动化测试范围，不定义产品功能。

不得自行增加新的需求入口或汇总文档。需求变化时直接更新上述固定文件。
